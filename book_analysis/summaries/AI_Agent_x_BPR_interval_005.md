## 2. ゴール設定と三層構造の組織設計
経営層のKGIをそのまま現場に降ろすだけでは現場は迷子になる。戦略の形骸化を防ぐためには、「トップダウン×ボトムアップ」の両輪により、目標を現場の行動に翻訳する**三層構造**の設計が必要である。

### 目標を実行に移す「三層構造」
- **経営層**: KGI（重要目標達成指標）を明確化し、「どんな成果を、いつまでに、どれほど出すのか」を示す。
- **変革リーダー層**: KGIを実現するための中間指標を設定し、経営目標を全社の具体的な計画に翻訳・横串管理する。
- **現場層**: 日々の行動指標（定量効果）を定義し、自身のユースケースで「どう貢献するか」を具体化する。

> **重要な成功定義**: 成功とは「AIが動いたら成功」「精度が高ければ成功」という技術的指標ではなく、「工数30%削減」や「リードタイム半減」といった**現場が体感できる定量効果**として最初に定義することである。

## 3. ユースケース選定のフレームワーク
ユースケース選定は「どの業務を自動化するか」ではなく「**どこで勝つか**」という戦略的判断である。投資対効果を見極める「選球眼」が求められる。

### 評価の4つの観点
ユースケースは以下の4観点でスクリーニングする。
1. **活用適性**: AIが強みを発揮するタスク（要約、抽出、照合、文章生成、データ分析など）を内包しているか。
2. **活用前提**: 実務上の土台（以下の5条件）が整っているか。
   - `精度要求レベル` / `業務非定型度` / `インプット整理度` / `業務標準化度` / `業務変更可否`
3. **想定導入効果**: 工数・時間・コストの削減規模。「人が判断しなくてもよい領域」をどれだけ広げられるか。
4. **拡張性**: 他部門や他業務プロセスに横展開（波及）できる要素を持っているか。

### 展開戦略
- **先行ユースケース**: 短期的に「クイックウィン（小さくとも明確な成果）」を出し、現場の空気を変える。
- **展開ユースケース**: 後続で類似業務へ波及させ、中期展開と全社的な投資対効果を両立させる。

## 4. ガバナンスとハイブリッド・チーム編成
多発型（複数ユースケースの並行進行）となるAIプロジェクトでは、「頭が多い状態」による意思決定の遅れを防ぐためのガバナンス設計が必須である。

### 推進体制と役割

- **PMO**: 進捗・課題・リスク管理。
- **業務改革リード（BPRコンサルタント）**: 「AIを前提に業務をどう変えるか」の中核を担い、業務と技術を繋ぐ。
- **業務担当者**: 実務ベースでのAs-Is分析や課題抽出。
- **技術担当者（AIエンジニア）**: 設計・実装と技術的助言。

> **リスク統制・ITインフラ管理（三層の外側のステークホルダー）**:

## 6. AIAM法（AI Agent Management）のフェーズ設計
AIプロジェクトの失速原因は技術的難易度ではなく「マネジメント不全（設計不全）」にある。AIAM法は、「AIをどう作るか」ではなく**「成果から逆算する」**ことを最大の特徴とする5つのフェーズで構成される。

### フェーズ1：基本計画立案（ルール作成）
- **目的**: 業務を「どう変えるか」を決める。
- **ポイント**: 対象業務の課題を明確にし、目指す業務成果（KPI）を関係者と合意形成する。「数字で成果を語る」状態を作る。

### フェーズ2：業務設計（BPRの実行）
- **目的**: 「AIならでは」の業務のあり方を設計する。
- **ステップ1**: As-Is分析（暗黙知や法令対応の前提を見落とさないよう深く分析）。
- **ステップ2**: To-Be設計（AIを前提として、以下の3つの問いに答える）。
  1. この業務の本来の目的は何か？
  2. 既存のプロセスは本当に必要か？
  3. 人が果たさねばならない役割は何か？
- **試行**: サンプルデータを用いてAIの限界（例：70%自動化、30%目視）を見極め、理想と現実のバランスを取る。

### フェーズ3：検証計画策定
- **目的**: 「どこまでやれば成功か（検証のゴール）」を定義し、終わりのないPoC（底なし沼）を避ける。
- **分離すべき2つの評価指標**:
  - `技術評価項目`: 正答率やエラー率など「業務の信頼性」を担保する指標。
  - `業務評価項目`: 工数削減率や処理件数など「業務における効果」を担保する指標。
- **リスク確認**: 法務・情報セキュリティ部門と連携し、機密情報や個人情報の取り扱い、AI特有のリスクについてお墨付きを得る。

### フェーズ4：構築・検証
- **目的**: 「使える」AIエージェントを作り上げる。
- **ポイント**: 構築→検証→改善→再検証のサイクルを**高速で回し**、業務の成果に直結させる。検証結果からネクストアクション（再設計やシステム開発移行）を判断する。

### フェーズ5：システム開発・導入
- **目的**: 実際の業務に組み込める「本番仕様」へ仕上げる。
- **ポイント**: 単なる便利ツールで終わらせず、既存の業務プロセスやシステムに組み込み、安全に利用できる状態でリリースする。
